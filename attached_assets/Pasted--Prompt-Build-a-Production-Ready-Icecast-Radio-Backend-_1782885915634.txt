# Prompt: Build a Production-Ready Icecast Radio Backend (Railway Compatible)

You are a senior backend engineer specializing in Python, FastAPI, Icecast, FFmpeg, yt-dlp, Linux, Docker, Railway deployments, scalable backend systems, and REST API design.

Your task is to build a **production-grade internet radio backend**. This is **NOT** a Discord bot, Telegram bot, or music bot. It is only the backend radio service.

I will build my own bot later using Python's `requests` library to communicate with your REST API.

The backend must be reliable enough to run on Railway 24/7 for months with minimal maintenance.

---

# Primary Goal

Build an always-online internet radio backend that:

- Streams audio to Icecast.
- Streams music directly from YouTube.
- Uses yt-dlp only for extracting direct media URLs.
- Uses FFmpeg for transcoding and streaming.
- Never downloads audio files.
- Uses a REST API for all controls.
- Automatically reconnects after failures.
- Never crashes because of one failed song.
- Automatically falls back to AutoDJ.
- Is optimized for Railway deployment.

# Streaming Pipeline

Strict pipeline:

YouTube

↓

yt-dlp extracts direct media URL

↓

FFmpeg reads directly from the media URL

↓

Icecast

Requirements:

- Never download media
- Never cache media
- Never save temporary files
- Never convert to local MP3
- Never write audio to disk

Everything must stream directly in memory using pipes.

# Playback Logic

The radio always operates in two modes.

Priority 1:

User Queue

Priority 2:

AutoDJ

Behavior:

If users add songs:

Play them after the current song.

When queue becomes empty:

Automatically return to AutoDJ.

The radio should never become silent.
